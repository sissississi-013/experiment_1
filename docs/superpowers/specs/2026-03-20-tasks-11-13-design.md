# Tasks 11-13: Tier 2/3 Tools, Dataset Resolver, End-to-End Run

**Date**: 2026-03-20
**Status**: Draft
**Approach**: C — LLM-Assisted Dataset Resolution + API-Based Tool Wrappers

## Overview

Extend the validation pipeline with three capabilities:

1. **Task 11 — Tier 2 Tool**: Roboflow-based object/content detection (API, free tier)
2. **Task 12 — Tier 3 Tool**: GPT-4o vision semantic quality checks
3. **Task 13 — Dataset Resolver + End-to-End Run**: Autonomous dataset acquisition and full pipeline validation on real COCO images

Design philosophy: API-based now for quick iteration, with the `BaseTool` wrapper interface designed so local models (GroundingDINO, LLaVA) can be swapped in later.

## 1. Dataset Resolver (New Pre-Step — Module 0)

### Purpose

Allow users to describe a dataset instead of providing a local path. The resolver interprets the description, figures out where to get the data, and downloads it — so the user never has to manually fetch images.

### Two-Phase Design

**Phase 1 — Resolution (LLM)**

GPT-4o receives the user's intent and outputs a structured `DatasetPlan`:

```python
class DatasetPlan(BaseModel):
    source: str              # "coco", "huggingface", "url"
    url: str                 # resolved download URL
    subset: str | None       # e.g., "val2017"
    category_filter: str | None  # e.g., "horse"
    max_images: int
    download_path: str       # where to save locally
```

**Phase 2 — Download (Deterministic)**

A `DatasetDownloader` class with source-specific handlers:

- `COCODownloader` — Uses COCO API (`pycocotools`) to fetch images by category/subset from the COCO dataset. Downloads annotation files (e.g., `instances_val2017.json`, ~240MB) to a local cache directory (`~/.cache/validation-pipeline/coco/`), filters by category, fetches matching image URLs.
- `HuggingFaceDownloader` — Uses `huggingface_hub` or direct URL to download image datasets from HuggingFace.
- `URLDownloader` — Generic HTTP download + archive extraction (zip, tar.gz). Handles direct image URLs or archive URLs.

Each handler takes a `DatasetPlan` and returns the local path to the downloaded images.

### Integration with Pipeline

- `UserInput` schema gets an optional `dataset_description: str | None` field
- If `dataset_path` doesn't exist and `dataset_description` is present, the resolver runs before Module 1
- If `dataset_path` already exists, the resolver is skipped entirely
- The resolver writes images to a local directory and updates `dataset_path` on `UserInput`

### New Schema

```python
# schemas/dataset.py
class DatasetPlan(BaseModel):
    source: str
    url: str
    subset: str | None
    category_filter: str | None
    max_images: int
    download_path: str
```

### New Module

```
validation_pipeline/modules/dataset_resolver.py
```

Contains:
- `resolve_dataset(description: str) -> DatasetPlan` — LLM call via instructor
- `download_dataset(plan: DatasetPlan) -> str` — dispatches to the right handler, returns local path

### Downloader Handlers

```

    coco.py          # COCODownloader

```

## 2. Tier 2 Tool — Roboflow Content Detection

### Purpose

Detect whether specific objects/content exist in an image. Answers content criteria questions like "does this image contain a horse?" that Tier 1 OpenCV tools cannot.

### Implementation

**New file**: `validation_pipeline/tools/wrappers/roboflow_wrapper.py`

**Class**: `RoboflowObjectDetectionTool(BaseTool)`

**Behavior**:
- Calls Roboflow's hosted inference API (free tier)
- Uses a pre-trained COCO model hosted on Roboflow Universe
- Accepts a `target_label` parameter to filter detections
- Returns confidence of the best-matching detection (0.0 if not found)

**`execute()` method**: `execute(image: PIL.Image, **kwargs) -> dict`
- `kwargs["target_label"]`: the object to look for (e.g., "horse")
- Returns raw detection dict (best confidence, all detections with bounding boxes)

**`normalize()` method**: `normalize(raw_output, calibration) -> ToolResult`
- `score`: confidence of best-matching detection (0.0 if absent, already 0-1)
- `raw_output`: full detection list with bounding boxes, labels, confidences
- `explanation`: human-readable summary (e.g., "horse detected with 0.87 confidence")

**YAML config**: `tools/configs/roboflow_object_detection.yaml`

```yaml
name: roboflow_object_detection
task_type: content_detection
subtask: object_detection
tier: 2
source: api
wrapper_class: "roboflow_wrapper.RoboflowObjectDetectionTool"
default_config:
  api_key_env: "ROBOFLOW_API_KEY"
  model: "coco/1"
cost_estimate_ms: 500
requires_gpu: false
```

**API key**: Read from `ROBOFLOW_API_KEY` environment variable. Stored in `.env` (gitignored).

## 3. Tier 3 Tool — GPT-4o Vision Semantic Check

### Purpose

Handle subjective/complex quality judgments that can't be reduced to a single numeric metric — "is the lighting natural?", "is this a professional photo?", "is the scene cluttered?".

### Implementation

**New file**: `validation_pipeline/tools/wrappers/openai_vision_wrapper.py`

**Class**: `GPT4VisionTool(BaseTool)`

**Behavior**:
- Takes an image + a semantic question (provided by the planner based on spec criteria)
- Sends the image to GPT-4o vision with a structured prompt asking for a 0-1 score and justification
- Uses `instructor` for structured output

**Structured output**:

```python
class VLMResult(BaseModel):
    score: float        # 0.0 to 1.0
    justification: str  # one-sentence reasoning
```

**`execute()` method**: `execute(image: PIL.Image, **kwargs) -> dict`
- `kwargs["semantic_question"]`: the criterion to evaluate (e.g., "natural lighting")
- Returns dict with `score` (0-1) and `justification` (string)

**`normalize()` method**: `normalize(raw_output, calibration) -> ToolResult`
- `score`: the LLM's 0-1 score
- `raw_output`: the full VLMResult dict
- `explanation`: the justification text

**YAML config**: `tools/configs/gpt4o_vision_semantic.yaml`

```yaml
name: gpt4o_vision_semantic
task_type: semantic_quality
subtask: vlm_check
tier: 3
source: api
wrapper_class: "openai_vision_wrapper.GPT4VisionTool"
default_config:
  api_key_env: "OPENAI_API_KEY"
  model: "gpt-4o"
  max_tokens: 100
cost_estimate_ms: 2000
requires_gpu: false
```

**Cost control**: Tier-based early exit means VLM calls only happen on images that already passed Tier 1 and Tier 2. Most rejections happen at cheaper tiers.

## 4. Pipeline Integration

### Updated Pipeline Flow

```
0. Dataset Resolution  (new, conditional — only if dataset_path missing)
1. Spec Generation
2. Calibration
3. Planning
4. Compilation
5. Execution           (now supports Tier 1/2/3 tools)
6. Supervision
7. Reporting
```

### Changes to Existing Code

**`BaseTool`** (`tools/base.py`):
- No signature change needed — `execute(image, **kwargs)` already accepts keyword args
- New Tier 2/3 tools use `**kwargs` for `target_label` and `semantic_question`
- Existing Tier 1 tools already ignore extra kwargs

**`UserInput` schema** (`schemas/user_input.py`):
- Change `dataset_path: str` to `dataset_path: str | None = None` (make optional for resolver flow)
- Add: `dataset_description: str | None = None`

**`PlanStep` schema** (`schemas/plan.py`):
- Add: `tool_params: dict | None = None` — step-specific parameters (target labels, semantic questions)

**`ProgramLine` schema** (`schemas/program.py`):
- Add: `tool_params: dict | None = None` — carried through from PlanStep

**Compiler** (`modules/compiler.py`):
- Copy `step.tool_params` into `ProgramLine.tool_params` during compilation

**Executor** (`modules/executor.py`):
- Pass `line.tool_params` through to `tool.execute(image, **(line.tool_params or {}))`
- Fix threshold comparison: currently does `raw_output >= threshold` which assumes a float. Tier 2/3 tools return dicts from `execute()`. Change to normalize first via `tool.normalize()`, then compare `tool_result.score >= threshold`

**Planner** (`modules/planner.py`):
- Already selects tools by `task_type`/`subtask` and tier
- Needs to populate `tool_params` for Tier 2 (target_label from content_criteria) and Tier 3 (semantic_question from quality/content criteria)

**ToolRegistry** (`tools/registry.py`):
- Add new entries to `WRAPPER_MAP` for `roboflow_wrapper.RoboflowObjectDetectionTool` and `openai_vision_wrapper.GPT4VisionTool`

### Error Handling for API Tools

Both Roboflow and GPT-4o tools are API-based and need resilience:
- **Retries**: Up to 3 retries with exponential backoff (1s, 2s, 4s) on transient failures (timeouts, 5xx)
- **Timeouts**: 10s for Roboflow, 30s for GPT-4o vision
- **Rate limiting**: Respect 429 responses with retry-after headers
- On permanent failure, return a `ToolResult` with `score=0.0`, `passed=False`, `explanation="API error: {details}"`
- The executor already catches per-image exceptions, so API failures degrade gracefully to "unusable" verdict

### New Dependencies

- `pycocotools` — COCO dataset API for downloading/filtering
- `huggingface_hub` — HuggingFace dataset downloads (optional)
- `requests` — HTTP calls for Roboflow API and dataset downloads

### Environment Setup

`.env` file (gitignored):
```
OPENAI_API_KEY=<existing>
ROBOFLOW_API_KEY=<your-key-here>
```

## 5. End-to-End Validation (Task 13)

### Test Scenario

**Intent**: "Find 50 sharp, well-exposed images of horses from COCO"

**Expected flow**:
1. **Resolver**: GPT-4o interprets → `DatasetPlan(source="coco", subset="val2017", category_filter="horse", max_images=50)`
2. **Download**: COCODownloader fetches 50 horse images from COCO val2017
3. **Spec**: Generates formal spec with quality criteria (blur, exposure) + content criteria (horse)
4. **Calibrate**: If exemplars provided, fit Platt scaling; otherwise use defaults
5. **Plan**: Tier 1 (blur + exposure) → Tier 2 (horse detection via Roboflow) → optional Tier 3
6. **Execute**: Per-image, early exit on tier failure
7. **Supervise**: Flag anomalies
8. **Report**: Final report with real verdicts on real images

### Test File

`tests/test_end_to_end.py` — Integration test that runs the full pipeline on real COCO data. Marked with `@pytest.mark.integration` so it doesn't run in normal test suite (requires API keys + network).

### Success Criteria

- Pipeline completes without errors
- `FinalReport` contains real verdicts (usable/recoverable/unusable) for all downloaded images
- All 3 tiers exercised (blur/exposure checks, object detection, optional VLM)
- Audit trail captures the full spec, plan, calibration, and supervision data
